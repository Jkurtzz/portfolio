To-Do List Full-Stack Web Application

Information

Purpose:

The purpose of this project is to learn and demonstrate the necessary skills of a full-stack engineer.

Overview:

This project was done to demonstrate the necessary skills of a full-stack engineer like setting up a database, creating a front-end user interface, and connecting the two with an api.
User authentication was implemented with hashed passwords, sessions, and Json Web Tokens.

Languages:

This app uses a MySQL database.
The api was written using Express.js.
The front-end was written using React.js.

Security and Authentication:

The app uses three levels of authentication. Using the Node.js crypto library, the user's information was hashed using the sha512 hashing algorithm. Passport-Local was used to serialize and deserialize users into sessions. Finally, Passport-JWT was used to sign and validate Json Web Tokens with an RSA key pair.

Data Flow:

General Process:

 Typically, the client makes an api fetch request with a payload stored in the body section and the client's JSON-Web-Token and Session ID stored in cookies. The API then uses middlware functions to verify the JSON-Web-Token and Session ID. If either the JSON-Web-Token or Session ID are expired or invalid, the api call is stopped, the client is sent back to the login page and their cookies are removed. If both pass, additional middlware functions are called to validate the payload. If any data is invalid, the api request is stopped and a failure message with the corresponding error is sent back to the client in the form of a json.
  Once all middlware functions have been passed, the api performs the appropriate HTTP method to the database and either retrieves, adds, edits or deletes data. If this HTTP method fails for any reason, a failure message with the corresponding error is sent back to the client in the form of a json. Upon failure, the front-end displays the error. If this HTTP method is successful, a success message and any requested data is sent back to the client in the form of a json. Upon success, the front-end reacts and changes depending on the newly added, edited, retrieved or deleted data. Any action on the front-end that involves adding, editing, or deleting a ToDo or user information will follow this process.

Sign-Up Process:

 When the client would like to make a new account, an API fetch request is made with the new user's information as the payload. The API then uses middleware functions to check if the username is available, if the password is at least 5 characters long and contains a non-alphanumeric character, if the two entered passwords match, and if any fields were left blank. If any of these checks fail, then the API request is stopped and a failure message with the corresponding error is sent back to the client.
 If all middleware functions have been passed, the entered password is then hashed using the Node.js crypto library. Then, a unique user ID is generated, and the user's ID, username, hashed password, first name and last name are stored in the database. Then, the user is "logged in" by adding this new user into a session, generating a session ID and a JSON-Web-Token, and then sending the JSON-Web-Token and Session ID back to the client in the form of a json.
 The client is then sent to their home page and stores both the JSON-Web-Token and Session ID as cookies. If the client makes any further api requests, it must present their JSON-Web-Token and Session ID, if either are missing or invalid, the user is sent to the login page and their cookies are removed.

Login Process:

 When the client tries to log in, an API fetch request is made with the username and password as the payload. The API then uses a middleware function to check if the entered username matches with a username in the database. If no username in the database matches the entered username, the API call is stopped and a failure message with its corresponding error is sent back to the client.
 If a username in the database matches the entered username, the entered password is hashed using the Node.js crypto library, and compared to the hashed password in the database. If these hashed passwords match, then the user is added to a session, a session ID and JSON-Web-Token are generated and sent back to the client in the form of a json. The client then stores this information in the form of cookies, which allows the client to verify itself, log in, and go to their home page.

Publication:

The app was published using Amazon Web Services (AWS). I purchased a domain name and received an SSL/TLS certificate for it. My app was then hosted on an EC2 instance and an Application Load Balancer connected the domain to the instance.
